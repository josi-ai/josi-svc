# D12 ‚Äî Real-Time Cosmic Event Streams

## 1. Purpose & Rationale

Astrology has peak-interest moments: eclipses, Mercury-retrograde starts, ingresses of slow planets (Saturn, Jupiter, Rahu/Ketu), conjunctions. Search volume spikes 10-50x on these days. Users want:
- Advance warning ("Saturn enters Pisces in 3 days ‚Äî here's what it means for your chart")
- Live context during the event ("Eclipse peaking now; read implications for your chart")
- Post-event reflection ("Last week's Mercury retrograde ‚Äî what changed?")

Today this is one-size-fits-all blog posts. Josi can deliver:
- **Per-chart real-time push** ‚Äî only notify you if the event touches your chart
- **WebSocket live channel** ‚Äî during eclipse hour, join a stream with your personalized minute-by-minute commentary
- **Enterprise/media API** ‚Äî partners consume cosmic event stream with timing + classical context

Differentiators:
1. Per-chart relevance (vs. generic "Mercury is retrograde!")
2. Rich classical context (citations; not cartoon astrology)
3. Live streaming UX on rare events (shared-moment engagement)
4. Multi-tradition coverage (Vedic + Western + Hellenistic reactions to same event)

## 2. Scope

### 2.1 In scope
- **Cosmic event catalog + scheduler**: all major events computed 5+ years forward (eclipses, planetary ingresses of Sun/Moon into signs, Saturn/Jupiter/Rahu/Ketu ingresses, stations of retrograde, exact aspects among slow planets, rare configurations)
- **Per-chart relevance engine**: for each event, compute which users' charts are "touched" (aspect within orb, house-significance trigger, transit to natal planet) and with what severity
- **Push notifications**: opted-in users receive timely push 72h / 24h / 1h / live for relevant events
- **WebSocket live channel**: during major events (eclipses, big ingresses), users can join a chart-personalized live feed
- **In-app live view**: dedicated UI during events; commentary generated by AI with citations (E11a)
- **Public events feed**: home-page widget for upcoming events (no personalization)
- **Enterprise API**: partners subscribe to event stream (webhook or SSE)
- **Silent mode**: users can pause notifications without disabling feature
- **Rate limits**: no more than N pushes per user per week regardless of event density

### 2.2 Out of scope
- General push notification platform (assumed already exists for app)
- Generic "daily horoscope" push (this is event-driven only)
- SMS delivery in P5 (push + email only)
- Video / audio livestream (text + voice via D1 is enough)
- Event prediction beyond astronomical certainty (astrology interpretation is rich; astronomy is exact)

### 2.3 Dependencies
- E6a Transit Intelligence already computes per-chart transit events
- F9 chart_reading_view for per-chart snapshot
- F12 prompt caching for cost-efficient per-chart commentary
- E11a AI chat for commentary generation
- S3 serving cache for live-view performance
- S5 durable workflow for scheduled push pipelines

## 3. Technical Research

### 3.1 Event catalog

Pre-computed 5 years forward, stored in `cosmic_event` table. Example event types:
- `eclipse_solar_total`, `eclipse_solar_partial`, `eclipse_lunar_total`, `eclipse_lunar_partial`
- `ingress_sign` (per planet √ó sign)
- `retrograde_station_start`, `retrograde_station_end`
- `exact_aspect` (Saturn-Jupiter, Saturn-Pluto, etc.)
- `nodal_ingress` (Rahu/Ketu sign change)
- `sadesati_start`, `sadesati_end` (per chart)
- `dhaiya_start`, `dhaiya_end` (per chart)

Astronomical timing from Swiss Ephemeris; verified vs. published almanacs.

### 3.2 Per-chart relevance engine

For a given cosmic event, relevance to a specific chart is computed:
- Does the event's planet currently aspect / transit a natal planet? (orb-controlled)
- Does the event's house (1-12 from transit-chart) affect this chart's significant house-lords?
- Is the event on the chart's dasa-period lord?
- Are yogas activated/deactivated by the event?

Output: relevance score 0-1 + activated factors list.

Threshold for push: score ‚â• 0.5. Users can tune sensitivity.

### 3.3 Scheduling and fan-out

At event-time minus N (72h, 24h, 1h, live), a worker fans out relevant pushes to eligible users. At 100M users, fan-out must be:
- Computed in shards (by user_id hash)
- Rate-limited per user per week
- Idempotent (duplicates elided)

Uses S5 durable workflow. Live WebSocket pub/sub uses Redis streams or comparable.

### 3.4 Cost management

Per-chart AI commentary on major events (100M users √ó eclipse = 100M LLM calls) is cost-prohibitive. Strategy:
- Templated commentary per event archetype (200 archetypes cover 95% of events)
- Per-chart personalization adds only the user's specific activated factors (small delta, cacheable)
- Prompt caching (F12) collapses common tokens
- Voice version (D1) available on-demand only; not pushed

### 5.2 Data model additions

```sql
CREATE TABLE cosmic_event (
    event_id             UUID PRIMARY KEY,
    event_type           TEXT NOT NULL,
    planets              TEXT[] NOT NULL,
    signs                TEXT[],
    exact_at_utc         TIMESTAMPTZ NOT NULL,
    window_start         TIMESTAMPTZ,
    window_end           TIMESTAMPTZ,
    visibility_regions   JSONB,                   -- for eclipses
    archetype_id         TEXT,                    -- maps to commentary template
    astronomical_details JSONB,
    created_at           TIMESTAMPTZ NOT NULL DEFAULT now()
);
CREATE INDEX idx_cosmic_event_time ON cosmic_event (exact_at_utc);

CREATE TABLE user_event_relevance (
    user_id              UUID NOT NULL,
    event_id             UUID NOT NULL REFERENCES cosmic_event(event_id),
    chart_id             UUID,
    relevance_score      NUMERIC(4,3) NOT NULL,
    activated_factors    JSONB NOT NULL,
    computed_at          TIMESTAMPTZ NOT NULL,
    PRIMARY KEY (user_id, event_id)
);

CREATE TABLE push_delivery (
    delivery_id          UUID PRIMARY KEY,
    user_id              UUID NOT NULL,
    event_id             UUID NOT NULL,
    stage                TEXT CHECK (stage IN ('minus_72h','minus_24h','minus_1h','live','post')),
    scheduled_at         TIMESTAMPTZ NOT NULL,
    sent_at              TIMESTAMPTZ,
    status               TEXT,
    suppressed_reason    TEXT
);

CREATE TABLE event_subscription_enterprise (
    subscription_id      UUID PRIMARY KEY,
    tenant_id            UUID,
    event_types          TEXT[],
    delivery             TEXT CHECK (delivery IN ('webhook','sse')),
    endpoint             TEXT,
    created_at           TIMESTAMPTZ NOT NULL
);
```

## 6. User Stories

### US-D12.1: As a user, I get a push 72h before Saturn enters my 10th house
**Acceptance:** push sent at T-72h; copy cites classical factor; tap-through opens chart view with context.

### US-D12.2: As an eclipse watcher, I join a live stream during the event tailored to my chart
**Acceptance:** WebSocket stream delivers per-minute frames; messages reference my activated factors; disconnect/reconnect preserves position.

### US-D12.3: As a casual user, I see upcoming public cosmic events on my home widget
**Acceptance:** widget lists next 30 days of major events; tap shows generic (non-personalized) explainer.

### US-D12.4: As a notification-averse user, I cap pushes to 2 per week
**Acceptance:** cap respected; suppressed pushes queue for next week or drop per user preference.

### US-D12.5: As an enterprise partner (media), I subscribe to eclipse webhook
**Acceptance:** webhook fires with event + archetype; delivery retried on 5xx; idempotency key respected.

### US-D12.6: As a regional user, I get eclipse pushes only if the eclipse is visible in my region
**Acceptance:** visibility filter applied; invisible eclipses flagged "not visible" vs. omitted per preference.

### US-D12.7: As ops, I can scale to 10M simultaneous WebSocket listeners during a global eclipse
**Acceptance:** load test; SLO ‚â• 99% message delivery P95 < 1s.
